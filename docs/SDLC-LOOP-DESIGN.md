# ai-sdlc-kit 全生命周期闭环开发 —— 设计思路详解

> 本文说明 **ai-sdlc-kit 现有方案是如何实现「AI 软件工程全生命周期闭环」的**。
> 面向想理解其内部机制、而非只会用命令的读者。读完你会清楚：闭环由哪几层拼成、
> 一个需求从进来到上线经过哪些产物流转、8 条方法论门如何被强制执行、以及环在哪里「闭合」、哪里仍是开环。

---

## 0. 一句话概括

> **用 gsd-core 当「全生命周期编排引擎」，把一套自有的《AI-SDLC 方法论》（8 条工程门）注入到引擎的规划/执行/验证环节，靠「阶段产物（artifact）按固定契约逐级流转 + 确定性门禁逐级把关 + 状态文件贯穿」形成闭环；codegraph 提供影响分析让「演进」也能回到环里，OpenHands 作为可选的重型执行器。**

闭环不是靠某一个「魔法命令」，而是靠 **三件事的叠加**：

1. **阶段化编排**：Discuss → Plan → Execute → Verify → Ship，每阶段有明确输入产物、输出产物。
2. **产物即接口**：上一阶段的输出文件，就是下一阶段的输入文件（`CONTEXT → PLAN → SUMMARY → UAT → TRACE`），环靠产物咬合。
3. **门禁即闸门**：8 条方法论门挂在「规划」「验证」「发布」三处，不过门就回流，形成「红 → 自愈 → 复验」的小闭环。

---

## 1. 三层架构：引擎 / 方法论 / 执行器

整个方案刻意分成三层，**各自独立升级、互不覆盖**，这是闭环能长期维护的前提。

```
┌─────────────────────────────────────────────────────────────┐
│  方法论注入层  overlay/   （自有资产，注入目标项目本地）         │
│   · CLAUDE.md 工程铁律段（8 条）                               │
│   · .claude/skills/sdlc-gates/（8 条机器可读规则）             │
│   · /sdlc-impact /gates /trace /release（4 命令）             │
│   · config 门控开关（verifier/security/gates）               │
│        ▲ 被引擎的 planner/verifier/plan-checker 读取并强制     │
├─────────────────────────────────────────────────────────────┤
│  引擎层  engine/gsd-core/  （上游精华，预构建，clone 即用）      │
│   · 69 命令 + 34 子代理 + 工作流 + 模板 + references + hooks   │
│   · Discuss/Plan/Execute/Verify/Ship 全生命周期编排            │
├─────────────────────────────────────────────────────────────┤
│  执行/索引层  executor/ + mcp/codegraph/                       │
│   · OpenHands（阶段4 重型执行器，Docker，可选）                │
│   · codegraph（MCP 代码知识图，影响分析/代码理解）             │
└─────────────────────────────────────────────────────────────┘
```

**为什么分层是闭环的基础**：
- 引擎（gsd）会随上游升级被 `integrate-engines.sh` 整包覆盖；
- 方法论是自有资产，放 `overlay/`，安装时注入**目标项目本地**（`.claude/` + `.planning/` + `CLAUDE.md`），**永不被引擎升级覆盖**；
- 两者通过 gsd 的**既有扩展点**对接（plan-checker 扫描 `.claude/skills/`、planner/verifier 读 `CLAUDE.md`、引擎读 `config` 决定门控开关）——即方法论不是改引擎源码塞进去的，而是「挂」在引擎留好的钩子上。这保证升级零摩擦，闭环不断裂。

## 2. 闭环主流程：一个需求的完整生命周期

下图是注入后，一个需求/功能从进入到上线再到演进的**完整环路**。方框是阶段，圆括号是该阶段产出的**产物文件**，菱形是**门禁**。

```
        ┌──────────────────────────────────────────────────────────┐
        │                                                          │
        ▼                                                          │
 ① /gsd-new-project ──► .planning/ 初始化 + 合并 config 门控开关       │
        │                                                          │
        ▼                                                          │
 ② /gsd-discuss-phase ──► CONTEXT.md（固化实现决策）                  │
        │   ◇ 需求质量矩阵门（7 类必覆盖：正常/异常/边界/权限/         │
        │     一致性/审计/兼容）—— 对抗式追问，人裁决                   │  ← 人工点①(需求)
        ▼                                                          │
 ③ /gsd-plan-phase ──► PLAN.md（按金字塔标注每个故事的测试层级）       │
        │   ◇ plan-checker 扫 .claude/skills/sdlc-gates/ 校验：       │
        │     测试金字塔 / 测试独立性 / 影响集 / 追溯 是否在计划里      │
        │   ◇ (演进时) /sdlc-impact 影响分析门：codegraph 定位调用方   │
        ▼                                                          │
 ④ /gsd-execute-phase ──► SUMMARY.md（波次并行执行，子代理产出）       │
        │   · orchestrator 协调，gsd-executor 子代理实际写码+提交     │
        ▼                                                          │
 ⑤ /sdlc-gates ──► 双验证门（确定性工具裁判，非 AI 自评）             │
        │   ┌─ 功能门：单元→集成/契约→API→E2E 逐层跑，收退出码        │
        │   └─ 非功能门：安全(SAST/越权)/性能(压测/N+1)/可维护性       │
        │                                                          │
        │   任一门红 ──► 自愈循环（≤3 次，每轮须有进展）──┐           │
        │        ▲                                       │           │
        │        └───────────────────────────────────────┘           │
        │   超 3 次 / 无进展 / 超预算 ──► 升级人工（附失败报告）        │  ← 人工点②(自愈超限)
        ▼   两门皆绿                                                 │
 ⑥ /gsd-verify-work ──► UAT.md（验收，verifier 守 CLAUDE.md 铁律）    │
        ▼                                                          │
 ⑦ /sdlc-trace ──► SDLC-TRACE.md（需求↔用例↔契约↔代码↔测试↔回滚）    │
        ▼                                                          │
 ⑧ /sdlc-release ──► 发布清单（渐进/回滚/开关/监控/双门绿证）          │
        │   ◇ 发布安全门：清单勾满 + 人工裁决签字才放行               │  ← 人工点③(发布)
        ▼                                                          │
 ⑨ /gsd-ship ──► PR + 归档                                         │
        │                                                          │
        ▼                                                          │
 ⑩ 演进/新需求 ──► 以 SDLC-TRACE 矩阵为索引，只重跑受影响部分 ────────┘
                     （变更增量传播，回到 ② 或 ③，闭环）
```

**环在哪里「闭」**：第 ⑩ 步是闭环的关键——发布后产生新需求或要改存量时，**不是从零重来**，而是以 `SDLC-TRACE.md` 追溯矩阵为索引，用 `/sdlc-impact`（codegraph 调用图）定位影响集，**只重生成/重跑受影响的那部分**，再回到规划阶段。这就是「变更增量传播」，让方案从「一次性流水线」变成「可持续演进的环」。

## 3. 闭环靠什么咬合：产物流转契约

闭环能转起来，本质是**每个阶段的产物，就是下个阶段的输入**，文件即接口。gsd 用工作流头部的 `loop-host` 注释声明这套契约：

| 阶段 | 命令 | 消费(输入) | 产出(输出) | 角色 |
|------|------|-----------|-----------|------|
| 讨论 | discuss-phase | 项目状态 | **CONTEXT.md** | orchestrator |
| 规划 | plan-phase | CONTEXT.md | **PLAN.md** | gsd-planner + gsd-plan-checker |
| 执行 | execute-phase | PLAN.md | **SUMMARY.md** | gsd-executor（波次并行）+ gsd-verifier |
| 验证 | verify-work | SUMMARY.md | **UAT.md** | orchestrator（对话式验收） |
| 追溯 | /sdlc-trace | 以上全部 | **SDLC-TRACE.md** | 方法论命令 |
| 发布 | ship / sdlc-release | UAT + TRACE + 双门绿证 | PR + 发布清单 | orchestrator |

贯穿全程的两个**状态文件**（让环跨会话不丢上下文，支持 `/clear` 后继续）：
- `.planning/STATE.md`：每阶段后更新，记录「现在到哪了」。
- `.planning/ROADMAP.md`：阶段列表与进度，`/gsd-autonomous` 靠它发现「下一个未完成阶段」。

> 正因为状态和产物都**落盘在 `.planning/`**（被 gsd 视为应入库的项目记忆），整个闭环是**可中断、可恢复、可审计**的，而不是一次性跑完即焚的脚本。

## 4. 编排机制：orchestrator + 子代理 + loop-host

这是闭环的「发动机」，决定了它能并行、能自愈、能跨会话。

### 4.1 编排器协调，子代理干活
`execute-phase` 的核心原则是 **「Orchestrator coordinates, not executes」**：
- 主编排器只做：发现计划 → 分析依赖 → 分波次（wave）→ 派发子代理 → 处理检查点 → 收集结果；
- 实际写代码、提交、产出 `SUMMARY.md` 的是 `gsd-executor` **子代理**，每个子代理加载完整的 execute-plan 上下文独立干活；
- **波次并行**：无依赖的计划分到同一波并行跑，有依赖的排到后续波。这让执行阶段能横向扩展，而主上下文保持精简（lean）。

34 个子代理各司其职（节选）：

| 子代理 | 职责 | 作用阶段 |
|--------|------|----------|
| gsd-planner | 从阶段范围生成详细计划 | Plan |
| gsd-plan-checker | 执行前审计计划质量（扫 skills 门禁） | Plan |
| gsd-executor | 执行计划任务、提交、产出 SUMMARY | Execute |
| gsd-verifier | 验证阶段完成度、把关质量门 | Verify |
| gsd-debugger | 诊断并修复问题（自愈循环主力） | Execute/Verify |
| gsd-security-auditor | 安全审计（非功能门一环） | Verify |
| gsd-codebase-mapper | 映射结构与依赖（影响分析支撑） | 全程 |
| gsd-nyquist-auditor | 校验「验证覆盖是否够采样密度」 | Verify |

### 4.2 loop-host hook：可插拔的扩展点
每个工作流文件头部声明 `loop-host`，定义了一组 **hook 触发点**（如 `execute:pre / execute:wave:pre / execute:wave:post / execute:post`）。这意味着闭环的每个关键节点都能被 hook 拦截/增强——方法论门、上下文纪律、自愈逻辑都挂在这些点上，而不必改工作流本体。

## 5. 方法论 8 门：闭环的「质量闸门」如何强制生效

这是 ai-sdlc-kit 相对裸 gsd 的**核心增量**。8 条门不是文档说教，而是通过 gsd 既有的三个消费点**强制注入到环里**：

| 注入物 | 装到哪 | 被谁消费（强制点） |
|--------|--------|-------------------|
| `sdlc-conventions.md` | `CLAUDE.md` 铁律段 | gsd-planner / gsd-verifier 强制读 `./CLAUDE.md` |
| `skills/sdlc-gates/rules/*` | `.claude/skills/` | gsd-plan-checker 自动扫描 `.claude/skills/` |
| `config 覆盖值` | `.planning/config.json` | 引擎读 config 决定 verifier/security/gates 开关 |
| `/sdlc-*` 命令 | `.claude/commands/` | 与 /gsd 共存，按需触发影响/门/追溯/发布 |

8 条门与其在环中的作用位置：

| # | 门 | 作用阶段 | 闭环意义 |
|---|----|---------|----------|
| 1 | **测试金字塔** | Plan/Verify | 计划阶段就为每个故事标注测试层级，禁止用 E2E 覆盖本该单元测的逻辑 → 保证测试套件可持续、闭环跑得快 |
| 2 | **双验证门(功能+NFR)** | Verify | 功能对不对 + 安全/性能/可维护性并行把关，**两门皆绿才放行** |
| 3 | **测试独立于实现** | Plan/Execute | 测试从「人确认的验收用例」生成，**禁止 AI 看自己代码补测试** → 防止「实现做什么测什么」的同义反复，保证门是真的能证伪 |
| 4 | **需求质量矩阵(7 类)** | Discuss/Plan | 对抗式追问把隐含需求/边界/并发/权限/兼容补全 → 闭环入口不漏需求 |
| 5 | **影响分析门** | Plan(演进) | 改存量前用 codegraph 调用图定位所有调用方，逐个评估 → **让「演进」能安全回到环里**（第 10 步闭合的技术支撑） |
| 6 | **自愈循环封顶** | Execute/Verify | 重试 ≤3、每轮须有进展、超预算转人工 → 给「红→修→复验」小闭环装刹车，防死循环烧资源 |
| 7 | **可追溯链** | 全程 | 需求↔用例↔契约↔代码↔测试↔发布↔回滚 全程不断链 → 变更增量传播的索引，闭环的「记忆」 |
| 8 | **发布安全** | Ship | 渐进发布+回滚就绪+特性开关+监控+人工裁决 → 闭环出口的安全阀 |

### 5.1 双验证门的关键设计：确定性裁判
门 2/5 最体现工程严谨：**「裁判是确定性工具（测试/扫描/构建），AI 不自评」**。
- 功能门：逐层运行**项目实际命令**（从 CLAUDE.md/config 读取），收集**退出码**判定绿红；
- 非功能门：SAST 无高危、依赖无高危漏洞、无密钥、越权负向测试、压测达基线、无 N+1、圈复杂度/重复率达标；
- **禁止为换绿而关闭鉴权/放宽校验/跳过门**——这条把「AI 偷懒糊弄过关」的路堵死，是闭环质量可信的根基。

### 5.2 自愈小闭环
门 6 定义了执行/验证里的**内层小环**：

```
跑双门 ──红──► gsd-debugger 修复 ──► 重跑双门
   ▲                                    │
   └──── 失败数下降/新增通过(有进展) ◄────┘
   超 3 次 / 无进展 / 超 token 预算 ──► 中止，升级人工（附失败报告+已试方案+阻塞点）
```

这是闭环里**唯一会自动转人工**的地方（除了三个预设人工点），保证 AI 不会无限空转。

## 6. 自动化程度与人工卡点

闭环是**「高度自动化 + 关键节点人工把关」**，不是全自动。`gsd:autonomous` 能把 Discuss→Plan→Execute 全阶段自动跑（`effort: max`），但**刻意保留人工点**：

**预设人工卡点（方法论 + 默认 config 决定）**：
1. **需求/原型审核**（门 4，Discuss）—— 对抗式追问后由人裁决需求是否完整；
2. **自愈超限升级**（门 6）—— 重试 3 次无果转人工；
3. **发布裁决**（门 8，Ship）—— 发布清单勾满 + 人工签字才上线；
4. 默认 config 还开了 `confirm_plan / confirm_breakdown / confirm_transition / issues_review` 等确认门，以及 `always_confirm_destructive / external_services`。

> 设计哲学：**把人放在「定义对错」（需求）和「承担后果」（发布）的节点，把「按定义干活」（编码/测试/修复）交给 AI 自动闭环**。这正是「双验证门 + 确定性裁判 + 测试独立性」三者合力的目的——让人不必信任 AI 的自评，只需信任确定性工具的绿灯。

## 7. 当前实现的边界：哪些是真闭环，哪些仍是开环

诚实标注，便于后续改造：

| 环节 | 状态 | 说明 |
|------|------|------|
| 需求→规划→执行→验证→发布 | ✅ 真闭环 | 产物流转 + 状态文件 + 门禁，完整且可恢复 |
| 演进/变更增量传播 | ✅ 真闭环 | TRACE 矩阵 + codegraph 影响分析支撑「只重跑受影响部分」 |
| 自愈（红→修→复验） | ✅ 真闭环 | 有封顶、有进展度量、超限转人工 |
| 双验证门的**裁判工具** | ⚠️ 半空 | 规则要求「跑项目实际命令」，但 SAST/压测/复杂度等**具体工具未内置**，需各项目自行接入；不接则 NFR 门形同虚设 |
| **生产反馈闭环** | ❌ 开环 | 发布后**无监控/可观测性数据回流到需求**；只有 mempalace（记忆宫殿）沉淀经验，不是真正的「生产 → 需求」回路。严格说这是「开发闭环」而非「软件工程全生命周期闭环」 |
| OpenHands 执行器 | ⚠️ 游离 | 仅 docker-compose 服务，**未与 gsd 命令流自动串联**，需手动启动喂任务 |

> 📌 这三个缺口的**开源解决方案与接入方式**见 [`SDLC-LOOP-GAPS-SOLUTIONS.md`](./SDLC-LOOP-GAPS-SOLUTIONS.md)。

## 8. 设计思路总结：闭环的 5 根支柱

把全文压缩成 5 句话，就是这套方案「让环转起来」的全部秘密：

1. **分层不耦合** —— 引擎可升级、方法论永驻本地，靠 gsd 既有钩子对接，升级不断环。
2. **产物即接口** —— `CONTEXT→PLAN→SUMMARY→UAT→TRACE` 文件咬合，阶段间不靠口头传递。
3. **状态可恢复** —— `STATE.md/ROADMAP.md` 落盘，环可中断、可续、可审计。
4. **门禁确定性** —— 8 门挂在规划/验证/发布，裁判用确定性工具，AI 不自评，红就回流自愈。
5. **演进有索引** —— TRACE 矩阵 + codegraph 影响图，让变更只传播到受影响处，把流水线弯成环。

> **本质**：它不是「让 AI 一口气写完一个项目」，而是「用工程化的产物契约和确定性门禁，把 AI 的每一步约束在可验证、可回流、可追溯的轨道上，从而让人只在『定义需求』和『批准发布』两端介入，中间全部自动闭环」。
